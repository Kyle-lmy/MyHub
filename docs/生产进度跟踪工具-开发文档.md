> 版本: v0.9 (确认班组自定义、节假日手动、修改记录PMC专属、Excel导入导出、并发提示)

## 一、背景

颐鑫纺织各部门间生产信息闭塞。订单录入后，裁剪、印花、刺绣、缝制各环节进度不透明，上下游无法提前获知准备信号，管理层无法监控全局进度。

### 现状

- 所有部门均反对填写数据，但公司流程化发展必须推行
- WPS 表格被否决——填起来太乱，缺乏输入约束
- Web 方案被否决——Kyle 不想再依靠 Web
- Kyle 一人开发和维护，担任 PMC 角色
- **同时在产 70-80 款**，每款 10-40 条 SKU 明细（色号×规格），总 SKU 数可达 700-3200

### 目标

**Python 程序 + SQLite 数据库文件，放网络共享文件夹，各部门运行同一程序读写数据。零服务器成本。PMC 统一下指令、排日程、管理日历班组、控制订单生命周期，各部门按指令填报每日数据。**

## 四、生产流程与数据流向

### 4.1 流程图

```
                  ┌──→ 印花 ──→ 退回裁剪 ──┐
需要公式 订单 → 裁剪(中枢) ─┤                        ├→ 裁剪发缝制 → 缝制 → 成品入库
                  └──→ 刺绣 ──→ 退回裁剪 ──┘
```

印花/刺绣完成后必须退回裁剪，由裁剪统一发缝制。不经过二次加工的，裁剪直接发缝制。

### 4.2 各部门每日记录

| 部门 | 每日记录字段 | 说明 |
|------|-------------|------|
| 裁剪 | ① 今日裁了 ② 发印花 ③ 发刺绣 ④ 收印花 ⑤ 收刺绣 ⑥ 发缝制 | 6 个数字/SKU |
| 印花 | ① 从裁剪接收 ② 今日完成 ③ 退回裁剪 | 3 个数字/SKU |
| 刺绣 | ① 从裁剪接收 ② 今日完成 ③ 退回裁剪 | 3 个数字/SKU |
| 缝制 | ① 从裁剪接收 ② 今日完成 ③ 成品入库 | 3 个数字/SKU |

### 4.3 结存逻辑（自动计算）

- 裁剪结存 = 累计裁了 - 累计发出(印花+刺绣+缝制) + 累计收回(印花+刺绣)
- 印花结存 = 累计接收 - 累计退回
- 刺绣结存 = 累计接收 - 累计退回
- 缝制结存 = 累计接收 - 累计入库

### 4.4 不记录不良数

废品/次品不单独记录。裁了 100 件就是 100 件，后续环节接收时以实际收到的数量为准，差异自然反映在结存中。

## 五、订单生命周期

### 5.1 状态流转

```
  [录入] → [待排程] → [进行中] → [已完成]
                         ↑              ↑
                    PMC 排程       PMC 标记完成
                                  (可部分完成+说明)
```

| 状态 | 触发 | 部门可见 | PMC 可见 |
|------|------|---------|---------|
| 待排程 | PMC 录入订单 | 不可见 | 可见（待办） |
| 进行中 | PMC 排程 | 可见（今天在排程期内 + 需要本部门） | 可见 |
| 已完成 | PMC 标记完成 | 不可见 | 可见（历史查询） |

### 5.2 排程

PMC 给订单设置排程日期范围（开始~结束日期），该期间内订单出现在部门视图中。
- 排程可以个别修改（延长/缩短日期）
- 今天是否在排程期内 + 今天是否为工作日（看日历），两者都满足才出现在填报列表

### 5.3 订单完成

- PMC 可标记完成，**允许部分完成**（如 900 件只完成 850）
- 标记时必须填写**完成说明**（如"客户取消剩余 50 件"或"尾数转下批"）
- 标记后订单从部门视图消失，PMC 可查历史

### 5.4 订单修改

- 订单信息可以修改（客户会变更订单：改数量、加色号、改交期等）
- **修改留痕**：记录修改人、修改时间、修改前/后值
- 已有流水数据不受影响（只影响后续填报的 SKU 列表）

### 5.5 每日数据修改

- 已提交的每日数据可以修改
- **修改留痕**：记录修改人、修改时间、修改前/后值
- PMC 和本部门操作员都可以修改

### 5.6 重复提交处理

- 同一天同一款同一部门第二次提交时：**累加**模式
- 弹出确认框："您今天已提交过 A001 的裁剪数据，本次提交将累加到已有数据上。确认不是重复提交？"
- 确认后新数据累加到当日已有数据上

## 六、日历与班组管理（新增核心模块）

### 6.1 为什么需要

工厂不是每天所有人都上班。周末和节假日哪些班组上班、哪些不上班，需要 PMC 统一管理。填报时只要求工作日的数据。

### 6.2 班组定义

新增"班组"字段，每个部门可以有一个或多个班组。**缝制部门有 50+ 班组**，班组列表不写死，PMC 可自行添加/维护。填报仍按部门维度，班组用于日历管理（哪些班组今天上班）。

**班组管理：**
- PMC 在日历模块中维护班组列表（增删改）
- 每个部门可关联多个班组
- 日历按班组粒度设置上班/休息
- 缝制 50+ 班组时，日历设置支持批量操作（如"全部缝制班组上班""指定班组休息"）

> Kyle 确认每个部门一个人填报，班组主要用于日历管理。

### 6.3 日历功能

PMC 可以在日历上设置每一天的工作/休息状态：

```
┌──────────────────────────────────────────────────────────┐
│  日历管理                                     2026年6月   │
│                                                          │
│  一    二    三    四    五    六    日                  │
│  ┌───┐┌───┐┌───┐┌───┐┌───┐┌───┐┌───┐                   │
│  │ 1 ││ 2 ││ 3 ││ 4 ││ 5 ││ 6 ││ 7 │                   │
│  │班 ││班 ││班 ││班 ││班 ││休?││休?│                   │
│  └───┘└───┘└───┘└───┘└───┘└───┘└───┘                   │
│  ┌───┐┌───┐┌───┐┌───┐┌───┐┌───┐┌───┐                   │
│  │ 8 ││ 9 ││10 ││11 ││12 ││13 ││14 │                   │
│  │班 ││班 ││班 ││班 ││班 ││休?││休?│                   │
│  └───┘└───┘└───┘└───┘└───┘└───┘└───┘                   │
│                                                          │
│  ── 6月6日(周六) 班组设置 ──                              │
│                                                          │
│  裁剪: [上班 ✓]  印花: [休息]                             │
│  刺绣: [上班 ✓]  缝制: [休息]                             │
│                                                          │
│  [保存] [设为每周重复]                                    │
└──────────────────────────────────────────────────────────┘
```

**功能说明：**

1. **工作日设置**：PMC 逐日设置每个班组是否上班（缝制 50+ 班组支持批量操作）
2. **节假日**：法定节假日由 PMC 手动设置（不内置中国节假日），一键设全员休息
3. **周末逐班确认**：每个周末逐个班组确认是否上班
4. **每周重复**：可设置某天的班次为每周重复模式
5. **填报约束**：只有"今天 + 本部门上班"的日子，才要求填报；休息日不催填
6. **班组维护**：PMC 可随时添加/删除班组（不写死，后续维护时自行录入）

### 6.4 日历对其他模块的影响

| 模块 | 影响 |
|------|------|
| 部门填报 | 只在"本部门上班日"显示待填订单，休息日不催填 |
| 填写监控 | PMC 看哪些部门该填没填，只看上班日 |
| 排程 | 排程日期范围内的工作日数量 = 实际可生产天数 |
| 图表 | 走势图只画工作日，休息日不画（避免断崖式下降假象）|

### 7.3 技术栈

| 层 | 选型 | 理由 |
|----|------|------|
| 语言 | Python 3 | Kyle 熟悉 |
| GUI | tkinter | 自带，表单+表格+日历控件 |
| 数据库 | SQLite + WAL | 单文件，零配置 |
| 打包 | PyInstaller | .exe 分发 |
| 可视化 | matplotlib | 6 种图表，嵌入 GUI + 导出 PNG |
| Excel 导入 | openpyxl | 通用导入：色号规格、订单、后续各种 |
| Excel 导出 | openpyxl | 数据导出 Excel |
| 备份 | shutil + schedule | 每周自动复制 data.db 到 backup 目录 |

### 7.4 并发写入处理

5 台电脑同时运行程序，同时读写共享 SQLite 文件。需要加并发提示机制：

- 程序定期（如每 30 秒）检查 SQLite 的锁状态
- 当某台电脑正在写入时，其他电脑提交时弹出提示："XX（操作员姓名）正在填写数据，请稍等片刻后重试"
- 配合 WAL 模式 + busy_timeout（5秒）+ 自动重试（3次）
- 极端情况下重试仍失败，提示"系统繁忙，请稍后再试"并保留已填数据不丢失

### 7.5 规模与性能考量

**同时在产 70-80 款 × 10-40 SKU/款 = 700-3200 条 SKU 同时跟踪。**

这对设计的影响：

| 问题 | 方案 |
|------|------|
| 看板一次显示 80 行太长 | 分页 + 搜索 + 按客户/状态/交期筛选 |
| 部门填报列表几十个款 | 搜索款号 + 折叠列表 + "只看未填"快捷按钮 |
| SQLite 查询性能 | 加索引（order_id, record_date, department, style_no） |
| 图表数据量大 | 图表按单款查看，不一次性画 80 款 |
| 甘特图 80 条 | 可滚动，按交期排序，超期置顶 |

### 7.6 数据备份

- 每周自动备份一次（程序启动时检查距上次备份是否超过 7 天）
- 备份文件：`\\server\生产跟踪\backup\data_2026-06-20.db`
- 保留最近 8 周备份，自动清理旧备份
